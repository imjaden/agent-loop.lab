# Agent Loop — 需求规范

## 一句话愿景

一个最小化的 AI Agent 循环框架。Agent 收到任务后自主执行 **Think → Act → Observe → Verify** 闭环，直到完成任务或预算耗尽。

---

## 核心理念

### 1. 自主推进

Agent 应能独立完成复杂任务，不需要人类每轮介入。人类只需给出任务描述，Agent 自动规划路径、调用工具、验证结果。

### 2. 循环闭环

```
用户任务 → [Think → Act → Observe → Verify]ⁿ → 最终答案
                ↑                              ↓
                └────── 不合格则重试 ←──────────┘
```

- **Think**：LLM 分析当前状态，决定下一步行动
- **Act**：执行工具（搜索、读写文件、运行代码等）
- **Observe**：收集执行结果，更新上下文
- **Verify**：检查输出质量，不合格则反馈改进

### 3. 安全兜底

Agent 可能失控或陷入死循环，必须有硬性停止条件：
- 迭代次数上限
- 运行时间上限
- 质量验证重试上限

## 功能需求

### A. LLM 通信层

| 需求 | 说明 |
|------|------|
| 消息发送 | 支持向 LLM 发送多轮对话上下文 |
| 工具调用 | 支持 LLM 返回工具调用指令 |
| 响应解析 | 正确解析 LLM 的文本回复和工具指令 |

### B. 工具系统

| 需求 | 说明 |
|------|------|
| 工具注册 | 可动态注册任意数量的工具 |
| 工具执行 | Agent 可调用已注册的工具并获取结果 |
| 内置工具 | 提供常用的 Web 搜索、文件读写、代码执行等工具 |

### C. 主循环引擎

| 需求 | 说明 |
|------|------|
| 循环控制 | 反复执行 Think → Act → Observe → Verify |
| 条件终止 | 输出答案满足质量要求时停止 |
| 强制终止 | 预算耗尽时强制退出 |

### D. 质量验证

| 需求 | 说明 |
|------|------|
| 自动检查 | Agent 输出答案后自动验证质量 |
| 反馈机制 | 不合格时注入改进建议，让 Agent 重试 |
| 重试上限 | 超过重试次数后放弃，返回当前最佳结果 |

### E. 状态管理

| 需求 | 说明 |
|------|------|
| 断点续跑 | 支持保存当前状态，中断后可恢复 |
| 状态恢复 | 从检查点加载历史上下文，继续执行 |
| 执行审计 | 记录每轮操作，便于回溯分析 |

---

## 非功能需求

### 可维护性

- 模块化设计：LLM 接口、工具系统、循环引擎相互独立
- 低耦合：通过依赖注入或构造函数组装各组件
- 易测试：核心逻辑可独立于 LLM 和工具进行单元测试

### 可扩展性

- 添加新工具：只需实现一个函数并注册
- 更换 LLM：符合 OpenAI 兼容协议即可
- 自定义验证器：可注入自定义的质量检查逻辑

### 健壮性

- 环境检查：API Key 等必要配置缺失时给出明确错误
- 超时处理：工具执行设置超时，防止卡死
- 异常捕获：工具执行失败不中断主循环，记录错误并继续

### 简洁性

- 单文件实现
- 最小外部依赖（仅 LLM SDK）
- 命令行友好，支持管道输入

---

## 输出规范

### 最终答案格式

- 结构化输出：优先使用表格、列表组织信息
- 先结论后细节：核心答案在前，详细说明在后
- 涉及日期的内容：应包含当前日期

### 文件命名规范

- 输出文件遵循格式：`{文档名称}-v{主版本}.{次版本}-{YYYYMMDD}.md`

---

## 设计约束

- 默认 LLM：deepseek-v4-flash（OpenAI 兼容）
- 默认运行环境：Python 3.12+
- 无需数据库或持久化存储（状态存文件即可）
